# 38장 브라우저의 렌더링 과정

## 핵심 용어

| 용어 | 설명 |
|------|------|
| **파싱** | 텍스트 문서의 문자열을 토큰으로 분해하고, 트리 구조의 자료구조(파스 트리)를 생성하는 과정. 파싱이 완료되면 파스 트리를 기반으로 바이트코드를 생성하고 실행 |
| **렌더링** | HTML, CSS, 자바스크립트로 작성된 문서를 파싱하여 브라우저에 시각적으로 출력하는 것 |

## 전체 렌더링 과정

```
① 리소스 요청/응답
   브라우저 → 서버에 HTML, CSS, JS, 이미지, 폰트 등 요청 → 응답 수신

② DOM / CSSOM 생성
   렌더링 엔진 → HTML, CSS 파싱 → DOM + CSSOM 생성 → 렌더 트리 결합

③ JS 실행
   JS 엔진 → JS 파싱 → AST 생성 → 바이트코드 변환 및 실행
   (DOM API로 DOM/CSSOM 변경 가능 → 렌더 트리 재결합)

④ 레이아웃 & 페인팅
   렌더 트리 → 레이아웃 계산 → 화면에 페인팅
```

---

## 38.1 요청과 응답

- 브라우저의 핵심 기능: 서버에 리소스를 요청하고 응답받아 시각적으로 렌더링
- 주소창에 URL 입력 → 호스트 이름이 DNS를 통해 IP 주소로 변환 → 해당 서버에 요청 전송

---

## 38.2 HTTP 1.1과 HTTP 2.0

| 버전 | 특징 |
|------|------|
| **HTTP 1.1** | 커넥션당 하나의 요청/응답만 처리 → 리소스 개수에 비례하여 응답 시간 증가 |
| **HTTP 2.0** | 여러 리소스 동시 전송 가능 → HTTP 1.1 대비 약 50% 속도 향상 |

---

## 38.3 HTML 파싱과 DOM 생성

> 서버가 응답한 HTML 문서는 문자열로 이루어진 순수한 텍스트.
> 브라우저가 이해할 수 있는 자료구조(DOM)로 변환해야 한다.

**DOM 생성 과정**

```
바이트(Bytes)
  ↓ (meta charset 기준 디코딩)
문자열(String)
  ↓ (어휘 분석)
토큰(Tokens)
  ↓ (구문 분석)
노드(Nodes)
  ↓ (부자 관계 반영)
DOM(트리 자료구조)
```

1. 서버가 HTML 파일을 메모리에 저장 → **바이트** 형태로 응답
2. meta 태그의 `charset` 기준으로 **문자열**로 변환
3. 문자열을 문법적 최소 단위인 **토큰**으로 분해
4. 토큰을 객체로 변환하여 **노드** 생성 (문서/요소/어트리뷰트/텍스트 노드)
5. HTML 요소 간의 부자 관계를 반영한 **트리 자료구조(DOM)** 구성

> **DOM은 HTML 문서를 파싱한 결과물**

---

## 38.4 CSS 파싱과 CSSOM 생성

- 렌더링 엔진이 `<link>` 또는 `<style>` 태그를 만나면 **DOM 생성을 일시 중단**
- CSS를 서버에서 로드 → HTML과 동일한 파싱 과정으로 **CSSOM 생성**

```
바이트 → 문자 → 토큰 → 노드 → CSSOM
```

- CSS 파싱 완료 후 중단 지점부터 HTML 파싱 재개

---

## 38.5 렌더 트리 생성

- DOM + CSSOM → **렌더 트리** 결합
- 렌더 트리는 실제로 **화면에 렌더링되는 노드만** 포함 (`display: none` 제외)
- 렌더 트리 → **레이아웃 계산** → **페인팅** 처리

> **리렌더링 주의**: 레이아웃 계산과 페인팅을 반복하는 리렌더링은 비용이 크므로 최소화해야 한다.

---

## 38.6 자바스크립트 파싱과 실행

- 렌더링 엔진이 `<script>` 태그를 만나면 **DOM 생성을 일시 중단**
- **제어권을 JS 엔진에 이전** → 파싱/실행 완료 후 렌더링 엔진에 반환 → HTML 파싱 재개
- DOM API를 통해 이미 생성된 DOM을 동적으로 조작 가능

### JS 엔진의 파싱과 실행 과정

```
자바스크립트 소스코드
        ↓
  ① 토크나이징(Tokenizing)
     소스코드를 어휘 분석하여 의미 있는 최소 단위인
     토큰(token)으로 분해
        ↓
  ② 파싱(Parsing)
     토큰의 집합을 구문 분석하여
     AST(Abstract Syntax Tree, 추상적 구문 트리) 생성
        ↓
  ③ 바이트코드 생성 & 실행
     AST를 기반으로 인터프리터가 실행할 수 있는
     중간 코드인 바이트코드(bytecode)를 생성하고 실행
```

| 단계 | 담당 | 설명 |
|------|------|------|
| 토크나이징 | 렉서(Lexer) | 소스코드를 토큰으로 분해 |
| 파싱 | 파서(Parser) | 토큰으로 AST 생성 |
| 바이트코드 실행 | 인터프리터 | AST → 바이트코드 변환 후 실행 |

> **AST(Abstract Syntax Tree)**: 소스코드의 문법 구조를 트리 형태로 표현한 자료구조.
> AST는 TypeScript 컴파일러, Babel, ESLint 등 다양한 도구에서도 활용된다.

## 38.7 리플로우와 리페인트

- JS 코드에서 DOM API를 사용하면 DOM 또는 CSSOM이 변경됨
- 변경된 DOM/CSSOM → 렌더 트리 재결합 → 레이아웃 및 페인팅 재실행

| 구분 | 발생 조건 | 설명 |
|------|-----------|------|
| **리플로우(Reflow)** | 레이아웃 변경 시 | 레이아웃 계산을 다시 수행 |
| **리페인트(Repaint)** | 항상 수반 | 렌더 트리 기반으로 다시 페인팅 |

> 레이아웃 변경이 없으면 **리플로우 없이 리페인트만** 실행된다.

---

## 38.8 자바스크립트 파싱에 의한 HTML 파싱 중단

- 브라우저는 HTML, CSS, JS를 **동기적**으로 파싱/실행 → `<script>` 위치가 중요
- `<script>`를 `<head>`에 두면 DOM이 완성되기 전에 DOM 조작 시 **에러 발생** 위험

**`<body>` 최하단에 `<script>`를 배치해야 하는 이유**

1. DOM이 완성된 이후 JS가 실행되므로 DOM 조작 에러 없음
2. JS 파싱/실행 전에 HTML 요소가 먼저 렌더링 → 페이지 로딩 체감 속도 향상

---

## 38.9 script 태그의 async / defer 어트리뷰트

> HTML5부터 추가된 속성으로, `src` 어트리뷰트로 외부 파일을 로드할 때만 사용 가능 (인라인 스크립트에는 적용 불가)

| 속성 | JS 로드 | JS 실행 시점 | 실행 순서 보장 |
|------|---------|-------------|--------------|
| 없음 | 동기 | 즉시 (파싱 중단) | O |
| **async** | HTML 파싱과 병렬 | 로드 완료 직후 (파싱 중단) | X |
| **defer** | HTML 파싱과 병렬 | HTML 파싱 완료 후 | O |

**async**
- HTML 파싱과 JS 로드가 **동시에** 진행
- 로드 완료 직후 즉시 실행 → 실행 순서가 로드 완료 순서에 따라 달라짐
- 순서 의존성이 없는 독립적인 스크립트에 적합

**defer**
- HTML 파싱과 JS 로드가 **동시에** 진행
- HTML 파싱이 **모두 완료된 직후** 실행 → DOM 생성 후 실행이 보장됨
- DOM 조작이 필요한 스크립트에 유용
