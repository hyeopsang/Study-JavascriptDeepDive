# 38장 브라우저 렌더링 과정

## 1. 파싱(Parsing)의 상세 과정

파싱은 단순한 텍스트 분석 이상의 복잡한 과정으로, 프로그래밍 언어로 작성된 텍스트 문서를 실행 가능한 형태로 변환하는 과정입니다.

- **어휘 분석(Lexical Analysis):** 문서의 문자열을 의미 있는 최소 단위인 **토큰(Token)**으로 분해합니다.
- **구문 분석(Syntactic Analysis):** 분해된 토큰들에 문법적 의미와 구조를 부여하여 트리 구조의 자료구조인 **파스 트리(Parse Tree)**를 생성합니다.
- **결과:** 생성된 파스 트리는 이후 중간 언어인 바이트코드로 변환되어 실제로 실행됩니다.

---

## 2. 렌더링(Rendering)의 전체 흐름

렌더링은 파싱된 문서를 실제로 화면에 픽셀 단위로 시각화하는 모든 처리 과정을 말합니다.

1. **리소스 로딩:** HTML, CSS, JS, 이미지, 폰트 등 모든 리소스를 서버에 요청하고 응답을 받습니다.
2. **파싱 단계:** 렌더링 엔진이 HTML과 CSS를 파싱하여 각각 **DOM**과 **CSSOM**을 생성하고, 이를 결합하여 **렌더 트리**를 형성합니다.
3. **자바스크립트 처리:** JS 엔진이 코드를 파싱해 **AST(추상 구문 트리)**를 생성하고 바이트코드로 변환해 실행합니다. 이때 DOM API를 통해 구조를 변경할 수 있습니다.
4. **레이아웃과 페인팅:** 렌더 트리를 기반으로 요소의 위치와 크기를 계산(Layout)하고, 실제 화면의 픽셀로 변환(Paint)합니다.

---

## 3. 요청과 응답

브라우저의 핵심 기능은 서버로부터 리소스를 받아와 시각적으로 표현하는 것입니다.

- **주소 입력:** 사용자가 URL을 입력하면 **DNS**를 통해 호스트 이름을 IP 주소로 변환하여 서버에 접속합니다.
- **기본 응답:** 루트 경로 접속 시 서버는 기본적으로 `index.html`을 응답합니다.
- **추가 요청:** HTML 파싱 도중 `link`, `script`, `img` 등 외부 리소스를 로드하는 태그를 만나면 파싱을 일시 중단하고 해당 리소스를 추가로 요청합니다.

---

## 4. HTTP 1.1 vs HTTP 2.0

| 특징          | HTTP/1.1 (1999)                | HTTP/2.0 (2015)                  |
| ------------- | ------------------------------ | -------------------------------- |
| **연결 방식** | 커넥션당 하나의 요청/응답 처리 | 하나의 커넥션으로 다중 요청/응답 |
| **주요 기술** | 순차적 전송                    | **멀티플렉싱(Multiplexing)**     |
| **성능**      | 리소스 개수에 비례해 속도 저하 | 1.1 대비 약 50% 이상의 속도 향상 |

---

## 5. HTML 파싱과 DOM 생성

바이트 형태의 데이터가 브라우저가 이해할 수 있는 객체 모델(DOM)로 변환되는 과정입니다.

1. **바이트(Bytes):** 서버로부터 이진 형식의 바이트 데이터를 전달받습니다.
2. **문자열(Characters):** 지정된 인코딩(예: UTF-8)에 따라 문자열로 변환합니다.
3. **토큰(Tokens):** 시작/종료 태그, 속성 등을 포함한 최소 의미 단위로 분해합니다.
4. **노드(Nodes):** 토큰들을 계층 구조를 가진 객체 노드로 변환합니다.

- _문서 노드, 요소 노드, 어트리뷰트 노드, 텍스트 노드_

5. **DOM(Document Object Model):** 노드들의 부모-자식 관계를 반영한 트리 구조를 완성합니다.

---

## 6. CSS 파싱과 CSSOM 생성

HTML 파싱 중 `link`나 `style` 태그를 만나면 CSS 파싱을 시작합니다. HTML과 유사하게 토크나이징 과정을 거쳐 **CSSOM(CSS Object Model)**을 생성하며, 이는 CSS의 **상속 관계**를 트리 구조로 명확히 반영합니다.

---

## 7. 렌더 트리(Render Tree) 생성

DOM과 CSSOM이 결합된 시각적 구조체입니다.

- **포함 대상:** 화면에 실제로 그려지는 요소 (예: `visibility: hidden` 포함)
- **제외 대상:** 시각적 요소가 아닌 태그(`meta`, `script`), `display: none`이 적용된 노드
- **레이아웃(Layout/Reflow):** 노드의 정확한 위치와 크기를 계산하는 과정입니다.
- **페인팅(Paint):** 배경, 테두리, 텍스트 등을 실제 화면 픽셀로 그리는 작업입니다.

---

## 8. 자바스크립트 파싱과 실행

자바스크립트 엔진(V8 등)은 제어권을 넘겨받아 다음 단계를 수행합니다.

1. **토크나이징:** 소스 코드를 의미 있는 토큰으로 분해합니다.
2. **파싱:** 토큰을 바탕으로 프로그램의 구조를 계층적으로 표현하는 **AST(Abstract Syntax Tree)**를 생성합니다.
3. **바이트코드 생성 및 실행:** AST를 인터프리터가 실행할 수 있는 바이트코드로 변환합니다. (V8 엔진은 자주 실행되는 코드를 기계어로 최적화함)

---

## 9. 리플로우(Reflow)와 리페인트(Repaint)

DOM이나 CSSOM 변경 시 발생하는 화면 갱신 작업입니다.

- **리플로우:** 레이아웃(위치, 크기)을 다시 계산하는 과정. (요소 크기 변경, 윈도우 리사이징 등)
- **리페인트:** 재계산된 레이아웃을 화면에 다시 그리는 과정.

> **주의:** 모든 리플로우는 리페인트를 동반하지만, 배경색 변경 등 레이아웃에 영향이 없는 경우는 리페인트만 발생합니다.

---

## 10. script 태그의 async/defer 어트리뷰트

자바스크립트 로딩에 의한 HTML 파싱 중단을 방지하기 위해 사용합니다. (외부 파일 참조 시에만 유효)

### 1) async 어트리뷰트

- **동작:** HTML 파싱과 JS 다운로드를 병렬로 진행합니다. 다운로드가 완료되면 즉시 실행하며, 이때 HTML 파싱은 중단됩니다.
- **특징:** **실행 순서가 보장되지 않습니다.** 의존성이 없는 독립적인 스크립트(분석 도구 등)에 적합합니다.

### 2) defer 어트리뷰트

- **동작:** HTML 파싱과 JS 다운로드를 병렬로 진행합니다. 실행은 **HTML 파싱이 완전히 끝난 후**에 이루어집니다.
- **특징:** **스크립트 작성 순서대로 실행**됩니다. DOM 조작이 필요하거나 의존성이 있는 앱 로직에 권장됩니다.
